# 의사결정 기록 (ADR)

> 추천 서비스를 만들며 내린 결정과 **그 근거(실측·시행착오)**를 남긴다.
> "왜 이렇게 했지?"를 나중에 되짚고, 같은 시행착오를 반복하지 않기 위함.
> 형식: 각 항목 = [결정] / [근거] / [버린 대안].

---

## 용어 정리 — 우리가 만드는 두 가지

- **수집 파이프라인** (데이터 엔지니어링): 추천의 재료(유저 즐겨찾기 등)를 로블록스에서 어떻게 모으나. 오늘 실측으로 다진 부분 대부분이 여기.
- **추천 알고리즘**: 모은 데이터로 어떤 게임을 추천할지 계산. cofavorite, 두 섹션, 유명도 보정 등.

# A. 로블록스 API Rate Limit (2026-07-04 규명)

## A-1. rate limit은 "동시성"이 아니라 "엔드포인트별 독립 토큰버킷"이다
- **결정**: 속도 제어를 "동시성 N"이 아니라 "초당 요청 수(rate)"로 하되, **rate 예산을 엔드포인트(API 경로)별로 따로** 관리한다.
- **근거 (측정 3단계에 걸쳐 결론이 바뀜)**:
  - 처음엔 "동시성 몇까지 되나"로 접근 → 결과가 들쭉날쭉 (멤버 동시성 20이 어떤 땐 0%, 어떤 땐 94% 에러). 원인: 순간 동시성이 아니라 "그 시점 토큰 잔량"이 변수.
  - 중간엔 "IP 전역 공유"로 오판 → 스레드 기반 테스트의 로컬 소켓 고갈이 만든 착시였음.
  - **최종 (asyncio+aiohttp 전면 재측정, full_api_test)**: rate limit은 **엔드포인트별 독립 버킷**. 서로 다른 API를 조합하면 처리량이 **합산(곱)**됨.
    - Phase2(5개 도메인 대표 동시): 각 API가 단독값 유지 → 도메인 독립.
    - Phase3(games 내부 fav+rec+detail 각 2/s): 총 6/s 성립 → **같은 도메인이라도 엔드포인트마다 독립 버킷**(도메인 합산 아님).
- **엔드포인트별 실측 지속 rate**: members 7.3 · icon 7.3 · media 7.2 · votes 6.9 · rec 6.2 · fav 6.1 · search 2.8 · place 1.6 · thumb 1.4 · explore 1.4 · username 1.2 · detail 1.1 · **grpinfo 0.3** (/s).
- **버린 대안**: (a)"동시성 N 고정" — 토큰 고갈되면 429 폭발. (b)"IP 전역 단일 버킷" — 측정 착시였음, 실제론 엔드포인트별.

## A-2. 429는 30초+ 페널티. 절대 밀어붙이지 않는다
- **결정**: 429 뜨면 즉시 rate 절반 + 백오프(2.5초~). 한계의 70~80%에서 보수 운영.
- **근거**: 429 유발 후 계속 찌르니 30초 넘게 회복 안 됨(페널티 누적). "빠르게 많이"보다 "느려도 꾸준히"가 총 처리량 높음.

## A-3. 실전 rate = 엔드포인트마다 AIMD, 목표는 측정치의 70~80%
- **결정**: 수집기에 AIMD(가산증가·곱셈감소) rate limiter를 **엔드포인트별로 하나씩** 둔다. 각 AIMD 목표 상한 = 그 엔드포인트 실측 rate의 70~80%(A-2 여유와 일치).
- **근거**: 엔드포인트별 독립 버킷(A-1)이라 리미터도 엔드포인트별이어야 함. AIMD가 동적으로 천장 아래를 탐침 → 정적 여유를 얇게 못박을 필요 줄어듦. collect_brainrot/jujutsu에서 실제 안정 동작(429 없이).
- **참고**: 병목은 grpinfo(0.3) · detail(1.1) · username(1.2). 이들은 rate를 올리는 게 아니라 **캐싱/한번만 호출**로 회피(→ F 섹션).

## A-4. 인증(.ROBLOSECURITY 쿠키)은 쓰지 않는다
- **결정**: 익명 호출만. rate 완화 위한 로그인 안 함.
- **근거 (웹 조사)**: 쿠키 자동화로 rate 넘기면 **계정 정지 위험**(ToS 위반, 행동 지문). 익명은 IP 일시차단(회복됨)이 최악이지만, 인증은 계정 밴 + 동일 IP로 메인계정까지 연관 추적 위험. 리스크 대비 이득 안 맞음.
- **버린 대안**: 부계정 쿠키 테스트 — 밴 위험으로 폐기.

# B. 수집 대상 선별 (2026-07-04)

## B-1. 대상 = "신생(2년 이내) 초인기작"
- **결정**: explore 차트 게임 중 created 2년 이내 + 그룹소유를 우선 수집.
- **근거**:
  - 인기 그룹게임 500개 중 **56%가 2년 이내** → 신생작이 인기 주류.
  - 신생작은 멤버가 적어 순회 짧음(rate 절약) + 즐겨찾기 수백만~수천만(코어 농도 높음).
- **"초인기" 기준**: 명시적 동접/방문 하한 없이 **explore-api 27개 차트 등재 여부**. (실전에선 동접 하한 추가 가능)

## B-2. 멤버는 Asc(오래된순)으로 수집
- **결정**: 그룹 멤버를 오래된 가입순(Asc)으로 순회.
- **근거**:
  - 오래된순 = 이 게임을 초기부터 판 코어층. 2년 이내 게임이라 **이탈 시간 짧아 초기 가입자도 여전히 활성**.
  - 실측: Steal a Brainrot 오래된순 200명 → 즐겨찾기 보유율 **95.6%**, 평균 26.6개. Jujutsu → 98%, 29.7개. (예전 RIVALS 뉴비 Desc는 6.6개, 부실)
  - Asc는 과거 가입자 순서가 고정 → **커서 밀림 없음** (Desc는 신규가입으로 밀림).
- **버린 대안**: Desc(최신순) — 뉴비라 즐겨찾기 부실 + 커서 밀림.

## B-3. 스킵 로직(뉴비/개발자) 전부 제거
- **결정**: 뉴비 스킵도, 개발자(앞 100명) 스킵도 안 한다.
- **근거**:
  - 뉴비 스킵: 신생 게임은 Asc로 초기 코어층부터 긁으니 불필요.
  - 개발자 스킵: 앞 100명 스킵 vs 미스킵 cofavorite 비교 → **거의 동일**(1~3위 완전 일치). dev 신호가 앞에 안 몰려있음(오히려 중간 구간에 많음). 앞 100 스킵해도 개발자 안 걸러짐.
- **버린 대안**: 위치 기반 스킵 — 효과 없음. (필요시 이름 [DEV]/[TEST]로 집계 단계 필터, 유저 스킵은 X)

## B-4. 차트 수집: explore 최우선(자주), Rolimons 후순위(보강)
- **결정**: explore-api 매일 재조회 + Rolimons 주1회 보강.
- **근거**: explore(공식, 744개)와 Rolimons(6,741개) 대조 → explore의 **22%(166개)가 Rolimons에 없음**(신생 급상승작 포함). explore가 최신 반영 빠름 → 우선. Rolimons는 롱테일 보강용, 갱신 느려 후순위.
- **버린 대안**: "Rolimons만 쓰기" — explore 고유 22% 놓침.

# C. 수집 표본 크기 (2026-07-04)

## C-1. 게임당 200명(1단계) → 커서 재개 500명(2단계)
- **결정**: 1단계는 게임당 200명 넓게, 유명작 커버 후 2단계로 커서 재개해 500명까지 깊이.
- **근거 (100명 단위 곡선 분석, 2게임 교차검증)**:
  - 새 게임 발견율: 100%→46%→26%→20%→13%... **500명쯤 포화** (두 게임 동일).
  - 겹침 신호(5+): 표본 클수록 계속 증가(포화 없음) → 상한 없지만 200명에서 이미 유효(Brainrot 98개, Jujutsu 217개).
  - **200명 = 최소 유효 표본**(상위 취향 확보). 애니 하드코어는 200명으로 신호 풍부, 양산 게임은 약간 부족 → 2단계로 보강.
  - Asc라 200명 지점 커서 저장 → 재개 시 밀림 없이 201~500 이어받기(중복 조회 0).
- **현실**: 게임이 너무 많아 **2단계까지 갈 일은 드물 것**. 대부분 200명에서 다음 게임으로.

# D. 추천 데이터 구조 (2026-07-04)

## D-1. "코어팬" 개념 폐기 → 즐겨찾기 무조건 저장
- **결정**: 그룹 가입 여부 무관. 조회한 유저의 즐겨찾기를 무조건 user_favorites에 저장. "팬" = 즐겨찾기한 사람.
- **근거**:
  - rate limit이 빡빡(하루 ~100만 요청) → 어렵게 조회한 유저 한 명도 못 버림.
  - 오래된순 그룹멤버 95~98%가 즐겨찾기 보유 → 그룹 가입 조건 빼도 손실 거의 없음.
  - 유저 기준 축적 → 한 번 조회가 여러 게임 cofavorite에 기여(크로스 재사용).
- **버린 대안**: "그룹+즐겨찾기 둘 다 = 코어팬"만 저장 — 조회 낭비 + rate 제약과 안 맞음.

## D-2. game_fans 테이블 삭제 → user_favorites 역조회로 통합
- **결정**: game_fans 삭제. "게임 X의 팬" = `user_favorites WHERE fav_universe_id=X`. 13개 → 12개 테이블.
- **근거**: 팬=즐겨찾기로 정의하니 game_fans와 user_favorites 역조회가 **같은 정보**(정규화 위반). idx_user_favorites_game 인덱스로 역조회 빠름. favorite_count는 COUNT로.
- **주의**: 아직 db-schema.sql 미반영. 다음 스키마 갱신 시 삭제.

## D-3. 실시간 그룹 조회 포기
- **결정**: 유저 요청(실시간)에서 그룹 멤버 순회 안 함. game_cofavorite 캐시만 조회, 없으면 건너뜀.
- **근거**: rate limit 페널티(30초+) 때문에 실시간에 그룹 순회(수백 호출)는 불가능. 팬 수집은 배치 전용.

# E. 추천 알고리즘 (2026-07-04)

## E-1. cofavorite = 장르 불문 실제 취향
- **결정**: 추천에 장르 개념 안 넣음. cofavorite(함께 즐겨찾기)만 믿는다.
- **근거**: Jujutsu(애니) vs Brainrot(양산시뮬) 교차검증 → **cofavorite가 장르 불문 취향을 정확히 반영**(애니 유저엔 애니게임, 양산 유저엔 양산게임). 같은 유저가 호러·RPG·레이싱 다 즐김 → 장르는 취향 경계가 아님. 다양성은 유저의 티어표 입력이 알아서 만듦.
- **버린 대안**: 장르 다양성 강제 — 취향 배신, 불필요.

## E-2. 추천을 두 섹션으로 (인기 / 발견)
- **결정**: 섹션1 = 유명도 보정 약함(검증된 인기작), 섹션2 = 보정 강함(숨은 취향작).
- **근거**: 정확도 vs 발견성 딜레마를 한 점수로 못 잡음 → 분리. 넷플릭스식 "인기 + 당신만의 발견".
- **공식**: `점수 = cofav / visits^alpha` (alpha 클수록 유명작 강하게 누름).

## E-3. 섹션2 필터: 방문 하한보다 "동접 하한"
- **결정**: 섹션2 후보 필터로 방문수 하한 대신 **동접(playing) 하한**(예: 100+).
- **근거**: 방문 하한(1000만)으로도 REMOVED/CLOSED/MAINTENANCE/베타가 남음(누적 방문은 과거값). **동접 하한이 "지금 살아있는 게임"을 정확히 거름**(죽은 게임은 동접 0). Jujutsu 실측에서 동접 필터가 훨씬 깨끗.
- **버린 대안**: 이름 필터([BETA]/[ALPHA] 제외) — 로블록스는 정식 게임도 [ALPHA]/[BETA] 딱지 흔함(Deepwoken 등). 거르면 진짜 인기작 놓침.

## E-4. 유명도 보정 강도(alpha)는 나중에 튜닝
- **현황**: alpha 0.15~0.6 실험. 0.15=약함(대작 안 눌림), 0.45+=과함(겹침 낮은 소형작 튀어오름). 0.25~0.35 근처가 균형이나 겹침 하한과 같이 조정 필요.
- **결정**: 실서비스 데이터 쌓인 후 튜닝. 지금 확정 안 함. (유명도 측정값도 visits/playing/favoritedCount 중 선택 여지)

# F. 실시간/배치 경계 · 캐싱 · rate 레인 (2026-07-05)

## F-1. 실시간은 username·fav 두 호출로 압축. 나머지는 전부 배치/캐시
- **결정**: 유저 요청(닉네임 검색→추천) 경로에서 로블록스를 실시간 호출하는 건 **username(닉네임→ID)**, **fav(그 유저 즐겨찾기)** 둘뿐. 게임 이름·설명·썸네일·cofavorite·차트·추천계산은 전부 배치로 미리 만들어 DB에서 조회.
- **근거**: 병목 API(detail 1.1/s 등)가 실시간에 있으면 동접 유저 몇 명만 몰려도 밀림. detail·icon·cofavorite를 배치 캐시로 빼면 실시간 경로에서 병목이 사라짐. "대부분 실시간"이 아니라 **"대부분 캐시 히트"**로 전환(rate limit이 이걸 강제).
- **버린 대안**: 요청마다 detail/rec 실시간 호출 — detail 1.1/s로 동시 유저 감당 불가.

## F-2. 게임 표시 데이터는 favoritedCount 상위부터 top-down 캐싱
- **결정**: games를 favoritedCount DESC로 정렬해 1위부터 detail(설명)+icon(썸네일)을 미리 수집·저장. 추천/표시에 뜨는 게임은 인기 상위로 편향되므로 상위 수천 개만 캐싱해도 대부분 커버.
- **순위 변동 처리**: "우선순위(무엇을 먼저 캐싱)"와 "신선도(캐시가 낡았나)"를 분리. 게임마다 detail 갱신 시점 기록 → ①캐시 없는 신규 진입은 새로 수집, ②오래된 것부터 재갱신(이름·설명·썸네일은 잘 안 바뀌니 주기 길게), ③favoritedCount 순위는 수집할 때 같이 갱신.
- **근거**: detail 내용은 거의 불변, 바뀌는 건 숫자뿐(표시용 근사치 OK) → 재갱신 주기 길게 잡아 병목 API 부담 최소화.

## F-3. fav는 "실시간 조회 + 무조건 저장", 자동갱신 없이 새로고침 버튼으로만
- **결정**: 유저 즐겨찾기는 캐시 있으면 **그대로 표시(자동 갱신 안 함)**. 새로 조회는 **새로고침 버튼**으로만, **사이트 접속당 1회**. 새로고침 누르면 우선순위 최후순위로. 조회 결과는 user_favorites에 무조건 저장.
- **근거**: fav는 유저 고유값이라 미리 캐싱 대상이 아님 → 요청 시 1회 조회가 맞음. 자동갱신을 끄면 재검색이 전부 DB히트(rate 소비 0). **실시간 fav 호출 1번이 (a)그 유저 추천 + (b)DB 확장(새 게임/유저를 collect_queue 등록) 두 몫**을 겸함(D-1 무조건 저장과 일치).
- **UX 구분**: 최초 조회(캐시 없음)=동기(로딩 대기, 화면 비어서), 새로고침(캐시 있음)=비동기(백그라운드 갱신).

## F-4. 진짜 구분은 "신규/기존"이 아니라 "username 필요 여부"
- **결정**: 조회 대기열을 두 레인으로. **username 레인**=진짜 신규(ID 없음)만. **fav 레인**=username 불필요한 나머지 전부(6개월+ 오래된 유저 및 새로고침).
- **근거**: 6개월+ 오래된 유저는 닉네임→ID가 이미 저장돼 username을 안 거침 → 신규가 아니라 fav 레인 최우선. 진짜 신규만 username(1.2/s 하드캡)을 소비.

## F-5. fav 예산 배분: 여유 10% · 신규 1.2/s · 나머지 4.3/s (work-conserving)
- **결정**: fav(6.1/s) 중 **여유 10%(0.6/s)** 예약, **신규 몫 1.2/s**(username 상한에 맞춤 — 더 줘도 신규는 못 씀), **나머지 ~4.3/s는 fav 레인**(6개월+ → 오래된 순).
- **근거**: 신규는 username(1.2/s)에 묶여 fav를 그 이상 못 씀 → 신규에 4.1/s를 줘도 2.9/s가 놀게 됨. 그래서 신규=1.2/s로 딱 맞추고 나머지는 대기열이 씀. 예약 비율은 고정 벽이 아니라 **최소 보장 바닥(floor)**으로, 신규가 안 쓰면 나머지가 넘겨씀(work-conserving 스케줄링).
- **여유 5% 기각**: 429 페널티 30초+ 대비 5%는 너무 얇음(측정 노이즈도 못 흡수). A-2(70~80% 운영=여유 20~30%)와도 충돌. **10%로 확정**(AIMD가 동적으로 추가 흡수하므로 20~30% 정적 여유까진 불필요).
- **닉네임→ID 캐싱**: 한 번 변환한 닉네임→ID는 DB 저장 → 재변환 없앰 → 신규 온보딩(username 병목) 부담 감소.

## F-6. 추천 계산은 cofavorite DB 탐색, depth1 우선(depth2는 부족할 때만)
- **결정**: 티어표 추천 계산 시 rec API 실시간 호출 안 함. 우리 game_cofavorite DB를 탐색. **depth1 우선**, 후보 부족할 때만 depth2로 확장.
- **근거**: cofavorite를 배치로 만든 이유가 실시간 rec 회피. DB 탐색이면 게임당 36개(depth2)도 그냥 DB읽기라 안 과함. 단 depth2는 한 다리 건너 취향이 멀어져 노이즈 가능 → depth1(직접 함께 즐겨찾기)만으로 티어표 여러 게임이면 후보 충분.

## F-7. 점수 = 티어표 여러 게임의 가중 합산 cofav (공식 골격은 유지)
- **결정**: 추천 점수의 입력을 "단일 게임 cofav" → **"티어표 여러 게임의 티어가중 합산"**으로. `raw(c) = Σ_g 티어가중치(g) × cofav(g, c)`. 그 뒤 유명도 보정·두 섹션·동접 하한은 E 섹션 그대로.
- **근거**: 예전 검증은 출발점이 게임 1개(Brainrot/Jujutsu)였으나 실서비스 출발점은 유저 티어표(여러 게임+순위). 공식(`cofav/visits^alpha`, 두 섹션)은 그대로고 **입력 만드는 법만 바뀜**.
- **티어 가중치 (확정, 단 값은 config JSON으로 조정 가능)**:
  - **SSS = 5.5** (특별 자리, 딱 2게임만 올릴 수 있음)
  - **A = 3, B = 2, C = 1**  (SS·S 등급은 없음)
  - **미배치 즐겨찾기 = 0.3** (약한 양의 신호)
- **역신호 안 씀**: 모든 가중치 양수. 낮은 등급도 "약한 선호"일 뿐 추천을 깎지 않음. 근거: **즐겨찾기 자체가 선호. 티어에 안 올린 건 싫어서가 아니라 배치가 귀찮았을 뿐** → 그래도 0.3의 양의 신호.
- **추천 후보에서 즐겨찾기 전부 제외**: 티어에 올린 것뿐 아니라 **미배치 포함 유저의 모든 즐겨찾기 게임을 추천 후보에서 뺌** (이미 아는 걸 추천하면 무의미). "신호 소스"와 "추천 후보"는 별개 역할 — 즐겨찾기는 신호로만 쓰고 후보에선 제외.
- **가중치는 config로**: 위 값들(및 alpha 등)은 코드에 박지 않고 config JSON에서 조정. rate 설정과는 별개 관심사라 **점수용 config 파일을 따로**(예: scoring.json) 두는 게 맞음.

# G. 2모드 · 나이보정 · 레인 확정 · 추가 실측 (2026-07-05 밤)

## G-1. 서비스 2모드 — Normal(빠름) / Precise(정밀·VIP)
- **결정**: 추천을 두 모드로. **Normal** = DB의 cofavorite만 사용(즉시). **Precise** = cofavorite 부족한 게임을 그 자리에서 실시간 팬수집해 보강(느림, VIP).
- **모드가 가르는 것은 딱 하나**: "추천 계산 시 cofavorite 없는 게임의 팬을 실시간 수집할 것인가". 그 외 전부(목록·검색·표시)는 모드 무관.
- **정밀 대상 등급**: SSS/A/B만 실시간 수집. C·D·미배치는 제외(비싼 수집을 낮은 신호에 낭비 안 함).
- **자격 미달 게임**(그룹 없음·오래됨 등)은 정밀모드로도 수집 불가 → 기존 신호만 안내.
- **근거**: 모드를 나눈 이유는 순수 API 제약. 실제로 비싼 건 팬수집(게임당 ~30초)뿐 → 모드는 여기에만 적용. 썸네일·장르(icon·detail)는 안 비싸서 모드 무관.
- **버린 대안**: 목록 표시(썸네일·장르)까지 모드로 제어 — icon·detail은 빠른 API라 모드로 아낄 실익 없음(개념만 꼬임).

## G-2. collect_queue 재정립 — "표시 정보 채우기"와 "팬 수집"을 분리
- **결정**: collect_queue = 표시용 게임 정보(detail·icon) 채우기 대기열. reason 컬럼으로 유입 구분(`easy_browse`/`precise_request`/`user_favorite`/`user_tier`/`recommendation`). 팬수집 대상 선정과 분리.
- **팬 수집 대상**은 큐가 아니라 games에서 자격으로 선별(그룹소유+신생+동접+fan_cacheable≠false). "유저가 넣은 게임에 200명 방식"을 강요하지 않음.
- **스키마 변경 불필요**: reason은 VARCHAR라 값만 추가. (필요시 priority INT 추가 여지, 지금은 requested_at+reason으로 충분)
- **근거**: collect_queue에 들어오는 게임은 그룹 없거나 오래됐거나 마이너일 수 있어 "그룹 오래된순 200명"이 안 맞음. 정보 채우기(조건 무관)와 팬수집(자격 필요)은 다른 일.

## G-3. 표시 정보(이름·장르·썸네일)는 lazy + b4 동반 저장. 전용 자동수집 없음
- **결정**: 표시 정보만 미리 긁는 독립 자동수집 잡은 **안 만듦**(아무도 안 보는 게임 미리 채우기 낭비). 대신 ①유저가 보는 게임은 lazy 실시간(icon·detail 빠름), ②b4 팬수집하는 게임의 detail·icon은 그때 같이 저장.
- **근거**: 추천에 나올 게임 = 팬수집 대상이니, b4가 정보도 확보 → 추천 목록은 대부분 DB에 있음. 그 외 마이너는 볼 때 lazy. 안 보고 추천에도 없는 게임은 아예 안 채움.
- **아이콘 특기**: search 응답엔 아이콘 URL 없음(실측) → universeId로 icon API 별도 1호출(100개/호출이라 목록 20개=1호출, 부담 0).

## G-4. 게임 필터는 "필터"가 아니라 3중 역할 분담 (동접·유명도·나이)
- **결정**: 오래된 게임을 칼같이 제외하지 않고 역할 분담:
  - **동접(playing) = 하드 필터**: 100 미만 제외("죽었나?" 이분법)
  - **유명도(visits) = 소프트 보정**: `/visits^alpha`("얼마나 인기?")
  - **나이(age) = 약한 보정**: 신선도(아래 G-5)
- **근거**: 오래됨 ≠ 죽음. Blox Fruits(6년, 동접 30만)는 살아있는 명작. 나이로 둘 다 누르면 명작이 억울. "죽음"은 동접이 정확히 잡고, 나이는 살아있는 것들 사이 미세 신선도만. 셋이 겹치지 않고 각자 다른 걸 잡음.
- **필터 속성 출처**: created·playing·favoritedCount·creator 전부 detail 응답에 포함(실측) → 별도 호출 0.

## G-5. 나이 보정 — 점(point) 방식, 약하게 (config)
- **결정**: 게임 나이별 점수 배수를 **점 리스트 + 선형보간**으로. 시그모이드 대신 값이 직관적인 점 방식.
  - points: `[3,1.0] [4,0.97] [5,0.92] [6,0.85] [7,0.78] [8,0.73] [9,0.68]`
  - 3년 미만 = 1.0, 점 사이 선형보간, 9년 초과 = 0.68 고정(바닥)
  - `age = 현재 − games.created`
- **적용**: `score = (raw / visits^alpha) × ageFactor(age)`.
- **근거**: "오래됐는데 동접필터·유명도보정을 다 견딘 게임 = 검증된 명작"이라 크게 누를 이유 없음. 나이는 약한 신선도 신호로 충분. 세 값 전부 scoring.json으로 튜닝 가능.
- **버린 초안**: 6년 0.45·7년 0.2(시그모이드 center 5.7) — 과함. 살아있는 오래된 명작을 반토막내서 폐기.

## G-6. rate 레인 구조 확정 (rate_governance.json)
- **결정**: 레인 3종 **realtime > precise > batch** 우선순위. floor=그 레인 최소 보장(호출/s). precise는 예약 없이 batch보다 우선(work-conserving — 안 쓰면 batch가 다 씀).
- **precise 전용 레인은 members·fav 2개만**: 정밀 팬수집의 핵심. detail·icon·grpinfo는 정밀도 기존 realtime/batch 공유(과분할 방지).
- **floor는 유저 대기 걸린 것만**: fav 1.2 / users_lookup 1.2 / icon 1.0 / detail 0.3 / place 0.3 / thumbnail 0.3. votes·media·rec·explore는 빠르거나 배치전용이라 floor 불필요.
- **margin 전부 12.5% 통일**: 부류별 세분(빠른10/느린15)은 과최적화 → 초기 통일, 부족하면 빠른 버킷부터 낮춰 튜닝. AIMD가 동적으로 추가 흡수.
- **근거**: 레인·floor·margin을 필요한 곳에만 최소로. 모든 버킷에 3레인 뿌리면 관리만 복잡. 실제 소비자만 레인으로.

## G-7. users_lookup 클린 재측정 (오염 제거 완료)
- **결정/실측**: GET단건+POST users+POST usernames **한 버킷 공유 재확인**(조합 시 합산 안 됨: GET 0.62+POST 0.50 ≈ 단독 1.38). 배치상한: **users(POST) 100 확정**(300=400), usernames 최소 100(150+는 rate로 400 못 봄). rate: GET단건 1.38/s, POST 각 0.75/s.
- **근거**: 지난 측정은 스캔 직후 부하로 429 오염 → 회복시간(측정 사이 40~90초) 넣어 격리하니 깨끗한 값. POST는 호출당 100개라 배치 유저처리량 압도적(rate 낮아도 무관).

- **search(omni-search)**: 응답에 universeId·name·playerCount·votes·description 포함, **아이콘 URL 없음** → icon 별도 보충.
