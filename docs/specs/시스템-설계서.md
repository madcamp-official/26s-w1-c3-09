# 1. 서비스 개요와 전체 구조

## 1-1. 서비스

로블록스 닉네임을 입력하면, 즐겨찾기와 티어표(취향 입력)를 바탕으로 맞춤 게임을 추천하는 웹 서비스.

페이지 4개:
```
① 닉네임 입력 → ② 티어표 배치(SSS/A/B/C) → ③ 추천 게임 목록(장르별 카드) → ④ 게임 상세(+영상)
```

## 1-2. 시스템 구조

```
[브라우저(프론트)] ←HTTP/JSON→ [웹 서버(백엔드)] ←SQL→ [MySQL (13테이블)]
                                    ↕ HTTP                    ↕ SQL
                              [로블록스 API]  ←HTTP→  [배치 워커 (Python, 상시 루프)]
```

| 구성요소 | 역할 |
|---|---|
| 프론트 | 4페이지 화면. DB에 직접 접근 불가 (반드시 웹 서버 경유) |
| 웹 서버 | 실시간 요청 처리. API 엔드포인트 8개 (E1~E8) |
| MySQL | 캐시 + 유저 데이터 + 운영 상태. 웹 서버와 배치의 **유일한 접점** |
| 배치 워커 | 주기 실행 수집기 (B1~B6). 웹 서버와 직접 통신하지 않음 |

- 로그인 없음. 닉네임 재입력으로 본인 티어표 재사용 (OIDC 도입은 보류 — 선택 기능)
- 배포: EC2 1대에 전부 (MySQL + 웹 서버 + 배치 + Nginx). 도메인으로만 접근 (IP 하드코딩 금지)

## 1-3. 설계 원칙

1. **캐시 우선**: DB에 있으면 API 안 부름. 없으면 즉석 호출 + 저장 (다음 사람은 캐시)
2. **lazy population**: 모르는 게임은 일단 ID만 기록(collect_queue), 상세는 배치가 채움
3. **실시간은 가볍게, 무거운 건 배치로**: 팬 수집(커서 순차) 같은 느린 작업은 절대 유저를 기다리게 하지 않음
4. **모든 연결은 불변 ID로**: universe_id / user_id (이름은 표시용, 언제든 바뀜)

# 2. 추천 알고리즘

## 2-1. 입력 — 티어표

- 재료: **유저 즐겨찾기 게임** (실시간 조회) + **유저가 검색으로 직접 추가한 게임**
- 유저가 SSS / A / B / C 4단 티어에 드래그 배치
  - **SSS는 최대 2개** (코드 검증)
  - **같은 티어 안에서도 왼쪽(position 낮음)일수록 더 좋아하는 게임** → 위치 보정에 사용
  - 미배치 게임: 추천 계산에서 제외 (또는 최저 가중치 — 파라미터)

## 2-2. 1단계 — 연쇄 추천 (기본 점수, 모든 티어 게임 대상)

People Also Join API(C-1, 게임당 최대 6개)를 재귀 적용:

```
티어의 각 게임 (시드)
 → depth 1: 연관 게임 6개          ← 캐시(game_recommendations) 우선, 없으면 API 호출+저장
 → depth 2: 각각 또 6개 (최대 36)  ← 동일. depth 1~2는 항상 보장 (API 호출 1+6=7회 이내)
 → depth 3: **캐시에 있는 것만** 확장 ← API 추가 호출 없음 (기회주의적 확장)
```

- depth가 깊을수록 가중치 낮음
- **다중 경로 도달(겹침) 시 소폭 가산**: 여러 갈래에서 닿는 게임은 연관성이 조금 더 높다고 봄. 단 "아주 약간"만 — 크게 주면 범용 인기작이 과대평가됨
- 이미 확장한 게임은 재확장하지 않음 (무한 루프 방지)
- depth 3가 캐시 상태에 따라 결과를 조금 바꾸는 비일관성은 **수용** (가중치가 낮아 영향 미미)

## 2-3. 2단계 — 팬 확장 (보강 점수, 최상위 게임만)

**티어표 최상위 2개 게임**(SSS 우선)에만 적용:

```
게임의 소유 그룹 (games.creator_group_id)
 → 그 그룹 가입 + 그 게임 즐겨찾기 둘 다 한 유저 = "팬"
 → 팬들의 즐겨찾기 게임 집계 (game_cofavorite, 배치가 미리 계산)
 → 겹친 팬 수(overlap_count)에 비례해 가중치
 → 겹침 1명뿐인 게임은 제외 (노이즈 컷)
```

- **실시간에는 캐시(game_cofavorite) 조회만** 함. 없으면 이 단계는 건너뛰고 collect_queue에 기록 → 배치가 채움 → 다음 요청부터 반영
- 팬 방식이 통하지 않는 게임 유형이 있음 (실측):
  - 틈새/커뮤니티 게임 (예: Korea Army): 팬 취향 뚜렷, 즐겨찾기 25~50개 → **잘 됨**
  - 초대중 게임 (예: RIVALS): 팬 대부분 즐겨찾기 1개, 겹침 0 → **무용** → `fan_cacheable=false`로 수집 스킵 (억누르려는 게 아니라, 수집해도 빈 결과라 배치 시간 낭비이기 때문)
- 2단계 점수는 1단계(유저 직접 취향)보다 낮게 보정

## 2-4. 최종 처리

1. 1단계 + 2단계 점수 합산 (게임별)
2. **유명도 보정**: visits/playing 기반으로 범용 인기작 점수 소폭 감산 (공식 미확정 — "유명한 데는 이유가 있다"는 관점에서 **약하게만**)
3. **제외**: 유저가 이미 즐겨찾기했거나 티어에 넣은 게임
4. 점수 내림차순 상위 N개 → 장르별로 묶어 표시

# 3. 가중치·점수 계산

## 3-1. 점수 공식 (구조 확정, 숫자는 튜닝 대상)

후보 게임 X의 점수:

```
score(X) = Σ (경로별 점수)  ×  겹침 보정  +  cofavorite 점수  −  유명도 보정

경로별 점수 = tier_w(시드의 티어) × pos_w(시드의 티어 내 위치) × depth_w(도달 깊이)

겹침 보정  = 1 + bonus × (도달 경로 수 − 1)     ※ bonus는 작게 (예: 0.1~0.2)
             또는: 첫 경로 100%, 추가 경로는 일부%만 합산

cofavorite 점수 = fan_w × overlap_count / sample_size   (2단계 대상 게임에서 나온 후보만)

유명도 보정 = pop_w × f(visits 또는 playing)             ※ 약하게. f는 log 스케일 권장
```

## 3-2. 파라미터 표 (전부 코드 상수 — DB와 무관, 언제든 변경 가능)

| 파라미터 | 의미 | 초기 후보값 (튜닝 전) |
|---|---|---|
| tier_w | SSS / A / B / C | 1.0 / 0.6 / 0.35 / 0.15 |
| pos_w | 티어 내 위치 보정 | 1위 100%, 이후 5%씩 완만 감소 |
| depth_w | depth 1 / 2 / 3 | 1.0 / 0.4 / 0.15 |
| 겹침 bonus | 다중 경로 가산 | 0.15 (아주 약간) |
| fan_w | cofavorite 계수 (1단계보다 낮게) | 1단계 최대치의 절반 이하 |
| pop_w | 유명도 감산 강도 | 약하게 (상위권 뒤집지 않는 수준) |
| N | 추천 개수 | 20 |
| 팬 임계값 | game_fans 등록 최소 즐겨찾기 수 | 10 |
| 팬 목표 | 게임당 수집 팬 수 | 20 (참고: Korea Army 20명 = 583 즐겨찾기) |
| 스킵 % | 그룹 최신 가입자 스킵 구간 | 미정 (10%는 과함 판정 — 축소 예정) |

**중요**: 이 값들은 전부 코드 설정입니다. 바꿔도 DB 스키마·저장 데이터는 그대로이며, 재계산만 하면 됩니다 (원시 데이터 user_favorites를 보관하는 이유).

## 3-3. 검증 방법

- 수집 완료된 실데이터(Korea Army 그룹 8,572명, 팬 212명)를 테스트셋으로 사용
- 파라미터 변경 → 재집계 → 추천 순위가 상식적인지 육안 검토 → 조정 반복

# 4. 실시간 파이프라인

유저 요청이 처리되는 흐름. 전부 웹 서버(E1~E8 엔드포인트)에서 수행.

## 4-1. 단계별 흐름

```
STEP 0  게임 직접 추가 (선택)
        검색어 → A-3(로블록스 검색, 실시간) → 유저 선택 → 티어표에 추가

STEP 1  닉네임 입력
        A-1: 닉네임 → userId (실시간, 없으면 "닉네임 없음")
        users UPSERT (첫 방문 INSERT / 재방문 last_seen_at 갱신)
        E-3: 즐겨찾기 조회 (항상 실시간 — 그 순간의 취향이어야 함)
        tier_entries 조회 → 재방문자면 지난 티어표 복원
        games에 없는 게임 → collect_queue 기록 (reason='user_favorite')

STEP 2  티어 배치 (프론트) → PUT /api/tiers
        SSS≤2 검증 → tier_entries 전체 삭제 후 재삽입 (유저당 1세트 덮어쓰기)

STEP 3  1단계 연쇄 추천 (§2-2)
        game_recommendations 캐시 우선 / 미스는 C-1 즉석 호출(재시도 포함)+저장

STEP 4  2단계 팬 확장 (§2-3)
        game_cofavorite 캐시 조회만.
        폴백(캐시 없음): (a) 이번엔 1단계만으로 응답 + (b) collect_queue 기록 — 동시 수행

STEP 5  점수 계산 (§3) → 정렬 → 상위 N
        user_recommendations 덮어쓰기 저장

STEP 6  결과 표시
        games에서 이름·장르·아이콘 부착 (없는 게임은 B-1 즉석 보충)
        상세 페이지 갔다 와도 user_recommendations로 복원 (재계산 없음)

STEP 7  게임 상세 (클릭 시)
        games + game_media 조회 (미스는 즉석 호출+저장)
        스크린샷: imageId → F-2로 URL 변환
        개발자 영상: video_asset_id → F-3(assetdelivery)로 재생 URL 매번 발급
        영상 없으면: game_videos 캐시 → 미스면 유튜브 G-1 (하루 100회 제한) → 저장
```

## 4-2. 동시성 관리 (전 이용자 공유 예산)

- 로블록스 rate limit은 **서버 IP 기준** → 모든 이용자 + 배치가 예산 공유
- **전역 세마포어**: 웹 서버의 로블록스 동시 호출 상한 50 (배치 몫 50과 합쳐 안전선 100)
- 세마포어 초과 요청은 자동 대기(FIFO) = **암묵적 대기열**. 몰려도 실패 대신 몇 초 지연
- 대기가 길어지는 것으로 확인되면(리허설 기준 5초+) → 명시적 대기열(jobId 발급 + 상태 폴링)로 확장. 처리 로직을 함수로 분리해두어 확장 비용 최소화. **DB 스키마 영향 없음** (메모리 큐)

# 5. 배치 파이프라인

Python 워커. 웹 서버와 통신하지 않고 **MySQL로만 연결**.
실행 방식(7/7 변경): b1→b2→b4 무한루프 상시 가동 — 기숙사 PC(PowerShell 루프)와 EC2(compose batch 컨테이너, restart:unless-stopped) 동일. 아래 표의 '주기(안)'은 초기 cron 설계의 흔적으로, 현재는 루프가 매 사이클 전부 실행한다.

## 5-1. 배치 목록과 주기

| ID | 배치 | 주기(안) | 하는 일 |
|---|---|---|---|
| B1 | 차트 수집 | 매일 1~2회 | 인기 게임 목록 확보 |
| B2 | 큐 소비 | 30분마다 | collect_queue의 게임 정보 채우기 |
| B3 | 추천 캐시 | 매일 새벽 | 인기 게임 C-1 미리 호출 |
| B4 | 팬 수집 | 매일 새벽 (분산) | 그룹 순회 → 팬 → cofavorite 집계 |
| B5 | 갱신·청소 | 매일 새벽 | 오래된 캐시 갱신, 만료 데이터 삭제 |

- 새벽 집중 배치(B3~B5)는 낮의 이용자 트래픽과 로블록스 호출 예산 경쟁을 피하기 위함
- 각 배치는 락 파일로 중복 실행 방지, 파일 로그 기록

## 5-2. B1 — 차트 수집

```
Rolimons gamelist (주력, ~6,700개, 브라우저 헤더 필수)
  실패(403 등) 시 → explore-api 26종 차트 폴백 (차트당 ~95개)
→ placeId → universeId 변환 (B-4, 변환 결과 재사용)
→ chart_snapshot 덮어쓰기 (이력 보관 안 함 — 최신만)
→ 새로 등장한 게임은 collect_queue 등록
```

## 5-3. B2 — collect_queue 소비 (lazy population 의 완성)

collect_queue에 기록되는 4가지 경우:
1. 유저가 마이너 게임을 티어에 배치 (cofavorite 없음)
2. 유저 즐겨찾기/직접 입력 중 DB에 없는 게임
3. 연쇄 추천 결과에 DB에 없는 게임
4. 실시간 수집이 시간 초과로 중단 (status='partial', 이어받기 대상)

처리:
```
pending 꺼냄 → B-1(50개 묶음)+B-2(투표)+F-1(아이콘) → games 채움 → done
실패 → failed (재시도 관리)
```
→ 서비스를 쓸수록 DB가 실제 유저 수요를 따라 저절로 채워짐

## 5-4. B4 — 팬 수집 (가장 무거움, 핵심)

대상: creator_type='Group'이고 fan_cacheable ≠ false인 게임 (차트 상위 + 큐 유입)

```
① E-0 그룹 정보 → member_count 저장 → "최신 가입자 X% 스킵 지점" 계산
② E-2 그룹 멤버 커서 순회 (Desc, 100명/호출, 순차 — 병렬 불가)
   - group_cursors.anchor_cursor 있으면 거기서 시작 (정착 유저 지점 재사용)
   - progress_cursor 있으면 중단 지점부터 이어받기
③ 멤버들의 E-3 즐겨찾기 병렬 조회 (동시성 ~50)
   - user_favorites에 이미 있고 최신이면 재조회 생략 (크로스 게임 재사용)
④ "그룹 가입 + 해당 게임 즐겨찾기" = 팬 판별
   - 즐겨찾기 수 임계값 이상만 game_fans 등록 (favorite_count, recorded_at 포함)
   - 팬들의 즐겨찾기 원본 → user_favorites (원시 보관 — 공식 변경 시 재계산용)
⑤ 목표 팬 수 도달 → 그 지점 커서를 anchor로 저장, collection_status='complete'
   중단 → progress_cursor + fans_collected 저장, status='in_progress' (다음에 이어서)
⑥ 집계: user_favorites에서 GROUP BY (직접 SQL)
   → 겹침 1명 제외 → game_cofavorite 저장 (overlap_count, sample_size)
⑦ 품질 판정: 팬 평균 즐겨찾기가 극소(RIVALS형)면 fan_cacheable=false → 이후 스킵
```

## 5-5. B5 — 갱신·청소

```
games: updated_at 24시간 초과 → B-1 재조회 (이름·동접 갱신 — 이모지 개명도 자동 반영)
       ※ updated_at은 로블록스 재조회 시에만 코드가 명시적 갱신
         (fan_cacheable 등 내부 변경으로 "신선한 척" 되는 것 방지 — ON UPDATE 자동갱신 안 씀)
user_favorites: recorded_at 1년 초과 → DELETE (오래된 취향 폐기)
game_recommendations / game_media / game_videos: fetched_at 기준 재수집
group_cursors: 오래된 앵커 재검증 (커서 밀림 대비)
```

전체 SQL은 `db-schema.sql` (MySQL 8.0, 전 테이블 utf8mb4·InnoDB). 여기는 요약.

## 6-1. 타입 원칙 (실측 근거)

| 원칙 | 근거 (실측 최대값) |
|---|---|
| 모든 ID → **BIGINT** | placeId 140조, universeId 102억, userId 73억, imageId 119조 |
| visits, favorited_count, member_count → BIGINT | visits 625억 |
| playing, votes, count류 → INT | playing 55만, votes 1,200만 |

| rank는 예약어 → rec_rank / chart_rank로 명명 | MySQL 8.0 |

## 6-2. 테이블 13개

**게임 캐시 (4)**
| 테이블 | PK | 핵심 컬럼 | 역할 |
|---|---|---|---|
| games | universe_id | name, genre_l1/l2, playing, visits, creator_group_id, fan_cacheable, updated_at | 게임 사전 (중심). 행이 있으면 완성된 정보 보장 |
| game_media | (universe_id, sort_order) | asset_type, image_id, **video_asset_id**, fetched_at | 스크린샷+개발자 영상. ⚠️video는 유튜브 아님(로블록스 에셋) |
| game_recommendations | (from, to) | rec_rank(1~6), fetched_at | People Also Join 엣지. depth는 저장 안 함(조회 횟수로 구현) |
| game_videos | (universe_id, youtube_video_id) | title, thumbnail_url, fetched_at | 유튜브 폴백 캐시 (영상 1개=1행, 할당량 보호) |

**팬 파이프라인 (4)**
| 테이블 | PK | 핵심 컬럼 | 역할 |
|---|---|---|---|
| user_favorites | (user_id, fav_universe_id) | recorded_at | **유저 기준** 즐겨찾기 원시 풀. 크로스 게임 재사용, 1년 후 삭제 |
| game_fans | (seed_universe_id, fan_user_id) | favorite_count, recorded_at | 게임별 팬 명단 (그룹+즐겨찾기 확인자) |
| game_cofavorite | (seed, related) | overlap_count, sample_size, computed_at | 집계 캐시. 원시(user_favorites)로 언제든 재계산 가능 |
| group_cursors | group_id | member_count, **anchor_cursor**(정착 시작점), **progress_cursor**(중단 지점), fans_collected, collection_status | 그룹당 1행. 앵커/진행 분리로 이어받기 지원 |

**수집 운영 (2)**
| 테이블 | PK | 핵심 컬럼 | 역할 |
|---|---|---|---|
| chart_snapshot | (sort_id, universe_id) | chart_rank, snapshot_at | 인기 차트. 덮어쓰기(이력 없음) |
| collect_queue | universe_id | reason, status(pending/partial/done/failed) | lazy population 대기열 |

**유저 (3)**
| 테이블 | PK | 핵심 컬럼 | 역할 |
|---|---|---|---|
| users | user_id (로블록스 ID 그대로) | username, last_seen_at | 이용자. 로그인 없음 |
| tier_entries | (user_id, universe_id) | tier(SSS/A/B/C), position | 마지막 티어표 (유저당 1세트 덮어쓰기) |
| user_recommendations | (user_id, universe_id) | score, rec_rank, created_at | 추천 결과 저장 (뒤로가기 복원·재방문 열람) |

- 모든 테이블에 시각 컬럼 존재 (신선도/청소/재검증 판단 기준)
- 인덱스: user_favorites(fav_universe_id) — 역방향 집계용 / collect_queue(status)

## 6-3. FK 정책 (중요 — 의도적 설계)

**FK는 3개만**: game_media→games, tier_entries→users, user_recommendations→users

| 기준 | 판정 |
|---|---|
| 부모가 항상 먼저 존재 보장됨 (users: 닉네임 입력 순간 생성) | FK 건다 ✅ |
| 부모(games)가 나중에 채워질 수 있음 (lazy population) | FK 안 건다 ❌ |

games를 참조하는 테이블들에 FK를 걸면 "아직 모르는 게임" 저장이 거부되어 lazy population과 충돌함.
FK 없는 곳의 정합성은 collect_queue + 배치가 책임 ("없는 게임은 반드시 나중에 채운다").
스텁 행(빈 games 행 선삽입 + FK) 대안은 검토 후 기각 — NOT NULL 완화, 모든 저장·조회 경로에 껍데기 처리 필요 등 복잡성이 더 컸음.

# 7. 테이블 관계와 데이터 흐름

## 7-1. 관계도 (텍스트)

```
                        ┌─ game_media          (FK ✅, 1:N)
                        ├─ game_recommendations (논리, from/to 둘 다)
                        ├─ game_videos          (논리)
        games ──────────┼─ game_cofavorite      (논리, seed/related 둘 다)
      (중심 테이블)      ├─ game_fans            (논리, seed)
                        ├─ chart_snapshot       (논리)
                        ├─ collect_queue        (논리)
                        ├─ user_favorites       (논리, fav_universe_id)
                        ├─ tier_entries         (논리, universe_id 쪽)
                        └─ user_recommendations (논리, universe_id 쪽)

        games.creator_group_id ──→ group_cursors.group_id (논리)

        users ──┬─ tier_entries          (FK ✅, 1:N)
                └─ user_recommendations  (FK ✅, 1:N)

        ※ user_favorites의 user_id는 "수집된 팬 유저"로, users(이용자)와 다른 집단 — 연결 아님
        ※ tier_entries·user_recommendations는 users와 games를 잇는 다리 테이블 (연결 2개씩)
        ※ "논리" = FK 없이 코드의 WHERE universe_id=?로 잇는 관계 (§6-3)
```

## 7-2. 데이터의 일생 (출처 → 저장 → 소비)

| 데이터 | 출처 (누가 만드나) | 저장 | 소비 (누가 읽나) |
|---|---|---|---|
| 게임 정보 | 로블록스 B-1/B-2/F-1 (B2·B5 배치, 실시간 미스 보충) | games | 웹 서버 전 화면, 유명도 보정 |
| 미디어 | B-3 (상세 첫 조회 시) | game_media | 상세 페이지 (영상은 F-3로 URL 재발급) |
| 연쇄 추천 엣지 | C-1 (B3 배치 + 실시간 미스) | game_recommendations | STEP 3 (depth 1~2 보장, 3은 캐시 한정) |
| 인기 목록 | Rolimons/explore (B1) | chart_snapshot | 배치 대상 선정, (선택) 인기 탐색 화면 |
| 팬 명단 | E-2×E-3 판별 (B4) | game_fans | 갱신 시 재조회 대상 목록 |
| 즐겨찾기 원시 | E-3 (B4) | user_favorites | cofavorite 재집계 (공식 변경 시에도) |
| 팬 집계 | B4의 GROUP BY | game_cofavorite | STEP 4 (실시간은 조회만) |
| 커서/진행 | B4 수집 중 | group_cursors | B4 재개·이어받기 |
| 수집 할 일 | 실시간 4경우 (§5-3) | collect_queue | B2가 소비 |
| 이용자 | 닉네임 입력 (STEP 1) | users | 재방문 인사, 티어 복원 |
| 티어표 | 유저 드래그 (STEP 2) | tier_entries | STEP 3~5의 시드·가중치 |
| 추천 결과 | STEP 5 계산 | user_recommendations | 뒤로가기 복원, 재방문 열람 |
| 유튜브 | G-1 (상세 첫 요청 시) | game_videos | 상세 페이지 폴백 영상 |

**저장하지 않는 것** (실시간 전용): 닉네임→ID 변환, 이용자의 즐겨찾기(요청 시점 취향), 게임명 검색 결과, 영상 재생 URL(만료 토큰 — video_asset_id만 저장하고 매번 발급)
